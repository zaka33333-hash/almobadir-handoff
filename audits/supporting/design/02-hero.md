# Hero (.hero3) — Forensic Audit

> Source files: `website/index.html` (lines 141–299), `website/assets/v3.css` (lines 120–241, 1138–1150), `website/assets/v3.js` (lines 72–112).
> Screenshots reviewed: `design-audit/screenshots/hero3-{1440,768,390}.png` and `full-1440.png`.
> Wave 1 facts re-used (not re-flagged): brand fonts 404 → Almarai/Tajawal fallback; component.css is dead code; 28 console errors are font 404s.

---

## Executive Summary

The hero is **ambitious but over-orchestrated**. It packs a topbar promise, an animated eyebrow, an SVG wordmark, a 4-piece tagline, a lede, a signup form, a 4-tile KPI panel with halo and live pip, a 6-chip channel rail, a trust block, and a 50-second financial ticker into a single viewport that already lives directly under a separate `.hdr3` header (which itself contains a ticker, a mega-menu, search, locale, and a CTA). The first viewport is doing the work of three. Stripe, Linear, and Apple all earned their hero polish by *removing*; this hero is composed by *addition*. The result is a layout that reads as "we have a lot to prove" rather than "we are inevitable."

Beneath the noise, the bones are good: the `1.45fr / 1fr` split is sensible, the color palette is restrained (six tokens, all referenced — no rogue hex values in the `.hero3` block save for two intentional gradient stops on the wordmark and tagline), the `prefers-reduced-motion` block is comprehensive, and the `clamp()` discipline is consistent. But the build also ships **dead CSS** for a `.hero3__topbar`, `.hero3__brand-mark`, `.hero3__nav`, `.hero3__locale`, and `.hero3__scroll-cue` that have no markup (lines 136–145, 235–239) — the brief lists these as part of the hero, but they don't exist in `index.html`. That is V2 vanity engineering: code written for a design that was never built or got moved into `.hdr3` without sweeping the dust.

Three issues hurt the most: (1) the SVG wordmark is rasterized text inside a `viewBox` with `font-family: Mostaqbali` — when the brand font 404s, the SVG renders the fallback font *inside* a fixed-aspect viewBox, which can stretch or letter-collide depending on substitute metrics and is not a true vector mark; (2) the wordmark has no `<h1>` semantics on the visible heading — the tagline is a `<p>` and `<h2>` is buried in the right panel; (3) the lede's bold `25 مليون مشاهدة شهرياً` repeats verbatim in the KPI panel three inches away (line 186 vs. line 229) — telling the same story twice in one viewport reduces both. Fix those three and trim two atmospheric layers, and this hero crosses from "trying hard" to "inevitable."

## Cross-cutting observations (not numbered findings, just context)

- **Token discipline is mostly good** — only ~6 literal hex/rgba values across 121 lines of hero CSS, and they're all in deliberate gradients. That's better than 95% of the codebases this skill reviews.
- **The `prefers-reduced-motion` block (line 1138–1150) is comprehensive** — every animated `.hero3__*` is in the kill list. Credit where due.
- **The grid 1.45fr / 1fr split is a solid choice for editorial+data heroes** — it's the same ratio FT Magazine uses on long-form. Don't change it.
- **The RTL handling is mostly correct** — `inset-inline-end`, `margin-inline-start`, `border-inline-start` are used appropriately throughout. No `left`/`right` hardcodes in the hero block.

## Vanity Engineering Score (Hero only)

- **RCR (Requirement-to-Complexity Ratio): 6/10** — The hero needs to communicate "we are an Arabic business+finance content network with 6 channels and 1.5M+ followers, sign up to our newsletter." That's a 3-element job: brand, social proof, CTA. The implementation ships 11 distinct content blocks plus 7 atmospheric layers plus 4 simultaneous infinite animations.
- **Top vanity findings:** #4 + #5 (dead CSS for unused topbar/scroll-cue), #8 (60s hue drift nobody sees), #11 + #13 (fake live data), #17 (always-on glow filter).
- **Kill cost to remove all V1 vanity:** ~3 hours.
- **The Hard Question:** *If you removed the channels rail, the trust block, and the ticker from the hero — keeping only eyebrow, wordmark, tagline, lede, signup, and panel — what would actually be lost? Probably nothing. Those three blocks are a second viewport's worth of content sharing the first viewport's space.*

## 200-word summary

The `.hero3` is over-built. The structural bones (1.45fr/1fr split, token discipline, RTL handling, reduced-motion comprehensive, signup focus-within state) are solid 8/10 work. But the hero packs 11 content blocks + 7 atmospheric layers + 4 infinite animations + a 50s static "live" ticker into one viewport that already sits below a separately-rendered `.hdr3` header — collectively too much. Three issues are critical: the SVG wordmark is rendered text (not vector path) so its quality depends on the brand font that doesn't load; the visible H1 is an `aria-label` on an SVG with no semantic plain-text heading, and the headline statistic "25 مليون مشاهدة شهرياً" is repeated four times in the same 900px viewport (eyebrow, lede bold, KPI tile, trust mono). Three V2 vanity issues: dead CSS for `.hero3__topbar`/`brand-mark`/`nav`/`locale`/`scroll-cue` (lines 136–145, 235–239) with zero markup using them; fake live ticker with hardcoded TASI/Brent/Aramco numbers; signup form that fakes success without any backend. Trim 2 atmospheric layers, fix the H1 semantic, kill the data lies, and this hero crosses from "trying hard" to Stripe-class. Estimated cleanup: ~3 hours.
